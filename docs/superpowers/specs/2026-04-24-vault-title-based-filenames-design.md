# Vault Title-Based Filenames Design

**Status:** design (pre-implementation)
**Date:** 2026-04-24
**Author:** chris + copilot

## Purpose

Change the vault backend so memory filenames are human-readable title slugs instead of opaque nanoid strings. The nanoid remains the stable identity (stored in frontmatter); the filesystem path becomes a derived, cosmetic property.

## Drivers

1. **Human-readable vault.** The current `n86khHlXf88S8Fq4i1NwT.md` filenames undermine the core design goal of an Obsidian-browsable knowledge base. Title-based filenames like `keep-claude-copilot-snippets-in-sync.md` are instantly meaningful in Obsidian's sidebar and OS file search.
2. **Stable identity.** Relationships, flags, audit trails, vector index, and MCP clients all reference memories by ID. That ID must not change when a file is moved or renamed.
3. **Safe manual editing.** Users can rename or move files in Obsidian without breaking cross-references. The system detects and flags inconsistencies rather than silently corrupting data.

## Scope

**In scope:**

- Title-based slug filenames for new vault memories.
- `VaultIndex` — extracted shared id→path index built from frontmatter at startup.
- File rename on title update (`git mv` + index update).
- Collision handling (short nanoid suffix on duplicate slugs).
- Path-consistency checking in consolidation (new `path_mismatch` flag type).
- Duplicate frontmatter ID detection.

**Out of scope:**

- Migration of existing nanoid-named files to title slugs. Existing files keep their current names; the system handles both styles transparently since the ID comes from frontmatter, not the filename.
- Changes to the pg backend, service layer, MCP tools, or API contract.
- Auto-correction of misplaced files (flag-only; user decides).

## Decisions

| Decision                | Value                                                    | Rationale                                                                |
| ----------------------- | -------------------------------------------------------- | ------------------------------------------------------------------------ |
| Filename derivation     | Full title slug, lowercase, hyphen-separated, ASCII-only | Maximum readability in Obsidian and OS file browsers                     |
| Rename on title change  | Yes, `git mv` to new slug                                | Keeps filenames meaningful over time                                     |
| Collision handling      | Append `-<4-char-nanoid>` suffix                         | `auth-setup.md` vs `auth-setup-k9pm.md`; deterministic, no counter drift |
| ID source of truth      | Frontmatter `id` field                                   | Path is derived; file can be renamed/moved without losing identity       |
| Scope source of truth   | Frontmatter `scope` + `workspace_id`                     | Directory structure is organizational; frontmatter is canonical          |
| Mismatch detection      | Consolidation engine raises `path_mismatch` flag         | Fits existing scheduled flagging pattern; no auto-correction             |
| Index architecture      | Shared `VaultIndex` class injected into all repos        | Avoids circular dependencies; single index, O(1) lookups                 |
| Existing file migration | None — old nanoid filenames work as-is                   | Non-disruptive; frontmatter ID is read regardless of filename            |

## Identity Model

### Before

```
Filename:     n86khHlXf88S8Fq4i1NwT.md
Frontmatter:  id: n86khHlXf88S8Fq4i1NwT   (redundant copy of filename)
ID source:    filename (via inferScopeFromPath)
```

### After

```
Filename:     keep-claude-copilot-instruction-snippets-in-sync.md
Frontmatter:  id: n86khHlXf88S8Fq4i1NwT   (sole source of truth)
ID source:    frontmatter (via VaultIndex, built at startup)
```

The nanoid is still generated by the service layer at create time (unchanged). The vault backend uses it as the internal stable reference but no longer encodes it into the filesystem path.

### Slug rules

- Lowercase, hyphen-separated
- ASCII-only (transliterate or strip non-ASCII characters)
- No truncation — full title is slugified
- On collision within the same directory: append `-<4-char-nanoid>` suffix

### Example paths

```
workspaces/agent-brain/memories/keep-claude-copilot-instruction-snippets-in-sync.md
workspaces/agent-brain/memories/auth-setup.md
workspaces/agent-brain/memories/auth-setup-k9pm.md    ← collision suffix
project/memories/global-style-guide.md
users/chris/agent-brain/local-dev-preferences.md
```

## VaultIndex

A new class extracted from the existing `Map<string, IndexEntry>` inside `VaultMemoryRepository`.

### Interface

```typescript
interface IndexEntry {
  path: string;
  scope: MemoryScope;
  workspaceId: string | null;
  userId: string | null;
}

class VaultIndex {
  // Build by scanning all .md files and parsing frontmatter for `id`
  static async create(root: string): Promise<VaultIndex>;

  // O(1) lookup
  resolve(id: string): string | null;

  // Called after creating a new memory file
  register(id: string, entry: IndexEntry): void;

  // Called after renaming a file (title change)
  move(id: string, newPath: string): void;

  // Called after archive/delete
  unregister(id: string): void;

  // Generate slug, check for collisions in targetDir
  slugForTitle(title: string, targetDir: string): Promise<string>;

  // All registered entries (for consistency checks, full scans)
  entries(): Iterable<[string, IndexEntry]>;
}
```

### Startup behavior

1. List all `.md` files under vault root (existing `listMarkdownFiles`)
2. For each file, parse frontmatter to extract `id` (lightweight — `gray-matter` on the YAML header only)
3. Infer `scope`, `workspaceId`, `userId` from directory structure (existing `inferScopeFromPath`, minus ID extraction)
4. Register `id → { path, scope, workspaceId, userId }` in the index map

### Error cases

- **Missing frontmatter `id`**: Skip file, log warning. Could be a non-memory markdown file that passed the directory filter.
- **Duplicate frontmatter `id`**: Keep first occurrence, log warning. Consolidation will flag both files.

### Injection

`VaultIndex` is created once during vault backend startup and injected into:

- `VaultMemoryRepository` (replaces its internal `Map`)
- `VaultMemoryFiles` (replaces per-call path scanning)
- Consolidation consistency checker (for path-mismatch detection)

## Filename Lifecycle

### On create

1. Service layer generates nanoid → passes `Memory` to vault backend
2. Vault backend calls `index.slugForTitle(memory.title, targetDir)`
3. Slug is checked for collision; suffix appended if needed
4. File written at `<scope-dir>/<slug>.md` with nanoid in frontmatter
5. Index updated via `register(id, entry)`
6. Git commit as before

### On title update

1. Vault backend detects title change in the update payload
2. Generates new slug via `index.slugForTitle(newTitle, currentDir)`
3. If slug differs from current filename:
   - `git mv oldPath newPath`
   - `index.move(id, newPath)`
   - Writes updated frontmatter + content to new path
   - Single commit covers both the rename and content change
4. If slug is identical (e.g., only capitalization changed in title): no rename, just update frontmatter

### On scope change

File moves to the new scope directory. The slug (filename) stays the same unless it collides in the new location, in which case a suffix is appended.

### Manual user edits in Obsidian

| User action                  | Effect                                                                                           | Recovery                                                                             |
| ---------------------------- | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------ |
| Rename file                  | No breakage — frontmatter `id` intact. Index picks up new path on next rebuild (server restart). | None needed                                                                          |
| Move file to wrong directory | Frontmatter scope disagrees with directory                                                       | Consolidation flags `path_mismatch`                                                  |
| Edit title in frontmatter    | Frontmatter title and filename diverge                                                           | Harmless — parser uses frontmatter title. Next API-driven title update re-slugifies. |
| Delete file                  | Memory disappears from index on next rebuild                                                     | Expected behavior                                                                    |

## Consistency Checking

### Integration point

The consolidation engine (`ConsolidationService`) gains a new check alongside its existing duplicate/stale/superseded checks.

### Path consistency check

For each memory file during a consolidation run:

1. Parse frontmatter: extract `scope`, `workspace_id`
2. Infer scope and workspace from directory path
3. Compare:
   - Frontmatter `scope` vs directory-implied scope
   - Frontmatter `workspace_id` vs directory-implied workspace
4. On mismatch → raise `path_mismatch` flag

### Duplicate ID check

During index build and consolidation:

1. If two files share the same frontmatter `id`, flag both as `path_mismatch`
2. Flag reason describes which files conflict

### New flag type

`path_mismatch` added to the `FlagType` union:

```typescript
type FlagType =
  | "duplicate"
  | "contradiction"
  | "override"
  | "superseded"
  | "verify"
  | "path_mismatch";
```

### Flag details

- **Severity**: `needs_review` (no auto-correction)
- **Reason**: Descriptive message, e.g.:
  > "File at project/memories/auth-setup.md has frontmatter scope 'workspace' (workspace_id: 'agent-brain') — expected under workspaces/agent-brain/memories/"
- **Resolution**: Agent presents the flag at session start and recommends either moving the file or updating the frontmatter. User decides.
